## Known Limitations
- **Blocker/decision noise**: Regex captures fragments from explanatory text, not just actual blockers. Produces noisy output like "s, next actions are stated explicitly in sessions" as a blocker.
- **Missing edit chunks**: Only 5 edit chunks detected vs. dozens actually created. Pattern needs refinement.
- **No file content recovery**: Reconstructs metadata (which files changed) but not edit chunk content.
- **Cross-project leakage**: Other projects' tasks (T52, T53, etc.) appear in graph; reconstructor filters to T1-T30 range.

## Notes

The memory-bank protocol uses specific file structures:
- `tasks/T*.md` — individual task files with status, nextAction, blocker
- `activeContext.md` — active tasks, recent decisions, files in flight
- `session_cache.md` — session state, context token estimate
- `edits/YYYY-MM-DD/HHMMSS-*.md` — edit chunks with scope and file list
- `edit_history.md` — chronological index of all edit chunks

Key patterns in session transcripts:
- Task references: "T19", "Task 19", "T20" (uppercase T + digits)
- Status words: "blocked", "in_progress", "complete", "pending", "paused"
- Blocker phrases: "blocked because...", "blocker:", "waiting on..."
- Next action phrases: "next:", "next action:", "TODO:", "need to..."
- Edit chunk phrases: "Created edit chunk", "edit chunk in edits/", "wrote edits/"
- File change patterns: tool call `write`/`edit` with file path, `exec` with git commit
